About

Domain-specific languages enable concise and precise formalization of domain concepts and promote direct employment by domain experts. Therefore, syntactic constructs are introduced to empower users to associate concepts and relationships with visual textual symbols. Model-based language engineering facilitates the description of concepts and relationships in an abstract manner.

However, concrete representations are commonly attached to abstract domain representations, such as annotations in metamodels, or directly encoded into language grammar and thus introduce redundancy between metamodel elements and grammar elements.

In this work we propose an approach that enables autonomous development and maintenance of domain concepts and textual language notations in a distinctive and metamodel-agnostic manner by employing style models containing grammar rule templates and injection-based property selection.

We provide an implementation and showcase the proposed notation-specification language in a comparison with state of the art practices during the creation of notations for an executable domain-specific modeling language based on the Eclipse Modeling Framework and Xtext.

ECSS compared

In summary, grammar-first, model-first, and our approach all end up using grammar as the artifact that dictates both structure as well as style of the generated language implementation.

However, in our approach, the structure and style of the language under construction is modelled by two artifacts: the metamodel that contains the structural semantics of the language under construction; and the style model that depicts the textual representation of the language under construction (i.e. metamodel dependent), any Ecore-based language (i.e. metamodel agnostic), or a combination of both.

Example Model in Default Notation

Curly-brackets layout is the default layout e.g. generated by the Xtext framework when supplying metamodels to it.

Further, the order of declaration that is generated by default, is single, or in other words, requires models (instances) of a space transportation service to follow an exact and predefined sequence of element occurrences.

Example Model in Desired Notation

Instead of using curly bracket-based layout, we want to have indentation-based layout that uses whitespaces as separators. Also, instead of only allowing a single order of declaration, we want our language to allow arbitrary order of declaration, i.e. considering models (instances) as valid even if they deviate from their occurrence in the grammar.

In addition to indentation-based layout and arbitrary order of declaration, we want features separated by comma (as this style requirement increases the size of grammars quadratically) and the keyword of classes that extend the class ‘NamedElement’ in domain-specific metamodels to require to be defined in uppercase letters (example of a metamodel-dependent style rule).

Whitespace Aware Style Model

Let’s have a look at a concrete style model for indentation-based layout. The main content of this style model is the rule whitespaceClassRule ensuring that class features are indented more than the root of the class. A rule is defined by a name, followed by a colon and the name of the rule template generator, which may extend an existing template such as classTemplate in this particular example.

Next, the rule definition may be followed by a double-column and list of rule groups to which it belongs to. The use of rule groups in a rule makes rule calls that belong to the used rule group be eligible to be called within the rule that specify the use of that particular rule group.

Next, the header of grammar rules is produced and more specifically a String that is composed of: the class name in the metamodel (i.e. class.name calls class.getName()), followed by the terminal “ returns “ followed by, again the class name in the metamodel, followed by column, opening bracket, class name, and closing bracket.

Then, if the name slot is set, then start the object definition from the return value produced by calling nameDistRules with parameter slot that is named name. On the other hand, if the slot named name is not set, then the object definition starts with the object's name which is represented by the variable classname;

Then, a rule may define calls to rule groups and in this case the rule group nameDistRules; a rule call refers to TemplateFunctionCall in the ECSS language metamodel and the parameters passed to the TemplateFunctionCall cause a SlotCall, e.g. ~name[0..1]

Subsequently, an indentation, followed by a the return produced by calling attributeDistRules by passing the content of the slot named other as parameter, followed by a dedentation; From the property rule on the bottom "slot(name,name)", we can see that features named "name" have a higher priority to be associated to the slot "name". Thus, the slot "name" only contains the feature named "name", while the slot "other" contains all the remaining features. The slot name only contains a single attribute, as the upper bound on the content length is zero, while the slot "other" can take up to 99 features.

What we have seen so far is metamodel-agnostic (i.e. no domain-specific element has been mentioned in the style model);

In contrast, NamedElement refers to a class in our space transportation services metamodel--this acts as a showcase that ECSS models may also define parts that are metamodel-dependent; If we remember the metamodel of our space transportation services language, we remember that all classes, except the root class and the class for physical properties, extend the class NamedElement Therefore, this rule applies to all these classes. And more specifically, what is done in this rule is that the value for the variable classname is defined by the result returned by the OCL expression rule.class.name.toUpperCase()

* {} is global and applies therefore to all elements (including NamedElement); Therefore, to make sure the style rule NamedElement+ has higher priority, we assign the priority of the global rule to 1.0 (i.e. a lower value)

Similar to the NamedElement+ rule, the global rule defines the value for the variable classname by the return value produced by evaluating the OCL expression; However, in this case, the OCL expression returns the class name without converting it to uppercase.

Arbitrary Order Style Model

For this we define the model arbitrary-order.ecss and import our previous style at the top by stating import “ws-aware.ecss” In this case we define a template named attributeTemplate and map it to the Java class AttributeXtendRule.

Then we define a template generator with name attributeGenTemplate that extends the template defined in the previous line.

As want to have all features separated by a comma, we cannot simply use normal unordered groups, but have to enumerate the features in a more complex way. And we do this by allowing an arbitrary first feature and then having an unordered group of all the remaining features, prefixed by the comma. This shows that even slight adaptations to the concrete syntax - whether to have features separated by a comma or not - can have dramatic effects on the Xtext rule.

Let’s have a look in more detail: In the outer for-loop we iterate through available features and create attribute rules for them, then (still in the outer for-loop) we directly use Java code to create a list of (all) features from which we remove the feature that we currently iterate through by the outer for-loop.

Then, create an inner for-loop that iterates through the remaining features (i.e. features excluding the feature being iterate by the outer for-loop).

In this inner for-loop, we return the necessary brackets connected by the ambersent symbol (i.e. resembling unordered groups in grammar) and prefix a comma before returning the attribute definition of the feature being iterated by the (this) inner for-loop.
